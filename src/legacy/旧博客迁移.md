# 旧博客迁移方案

之前的博客是 [jackhai9.github.io](https://jackhai9.github.io/)，当前在用的是这个更轻量的 `blog`。这次迁移的目标不是替换旧站，而是把旧站内容复制一份到新站，同时尽量保留文章当时的原始信息，比如发布时间、分类、标签、原文地址。

旧站继续保留在线，内容和链接都不删。新站只负责承接和整理，后续如果需要再逐步把旧文挂到首页或做专题归档。

## 为什么不能直接手工复制

当前仓库在提交 Markdown 时会自动补时间：

```bash
CREATION_DATE=$(git log --diff-filter=A --follow --format=%ad --date=short -1 -- "$FILE")
LAST_UPDATE_DATE=$(date +"%Y-%m-%d")
```

如果直接把旧文拷进来再提交，旧文的 `最后更新日期` 会被改成迁移当天，这会把“迁移时间”和“原始发布时间”混在一起。正确做法是：

1. 迁移脚本只写旧文的原始信息
2. 不预填 `本文创建日期` 和 `最后更新日期`
3. 让当前仓库的 pre-commit 在首次提交时补迁移当天

这样一来，原始发布时间和当前仓库时间语义就不会打架。

## 迁移规则

1. 原始发布时间来自旧 Hexo 文章 front matter 的 `date`
2. 原文分类和标签来自 `categories`、`tags`
3. 原文地址按旧站 permalink 规则恢复
4. 正文中的 `<!--more-->` 会被自动删除
5. 极少数缺少协议头的裸链接会被自动修复
6. `layout: photo` 这类旧 Hexo 文章会转成普通 Markdown 图片列表

## 脚本位置

- 迁移计划：[`MIGRATION_PLAN.md`]({{ site.repo_url }}/blob/main/MIGRATION_PLAN.md)
- 迁移脚本：[`scripts/migrate_hexo_posts.py`]({{ site.repo_url }}/blob/main/scripts/migrate_hexo_posts.py)

## 使用方式

先预览，不写文件：

```bash
cd blog
python3 scripts/migrate_hexo_posts.py --dry-run
```

导出单篇样例到临时目录：

```bash
cd blog
python3 scripts/migrate_hexo_posts.py \
  --apply \
  --include 转投hexo-使用hexo在github写博客 \
  --output /tmp/blog-migration-sample
```

确认样例没问题后，再全量导入：

```bash
cd blog
python3 scripts/migrate_hexo_posts.py --apply --write-index
```

默认会导入到 `src/legacy/`，不会覆盖当前 `src/` 下已经在写的新文章。

## 迁移后保留什么

每篇导入文章底部都会保留：

- 原始发布时间
- 原文分类
- 原文标签
- 原文地址
- 原始来源文件

当前仓库自己的 `本文创建日期` 和 `最后更新日期` 不由迁移脚本写入，而是继续交给现有 pre-commit 补齐。这样做的好处是，原始信息和当前仓库时间分层清楚，不会互相覆盖。

## 迁移顺序

建议顺序很简单：

1. 先审核 `MIGRATION_PLAN.md`
2. 再审核迁移脚本
3. 先跑单篇样例
4. 再做全量导入
5. 最后把归档入口挂到首页

整个过程本质上是“复制迁移”，不是“替换迁移”。旧站继续保留，新站逐步接手内容展示和整理即可。
